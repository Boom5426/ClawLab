好，先不追求“大而全”，我们直接把 **ClawLab** 收敛成一个真正能做出来的最小可闭环 MVP。

我会按这条主线来定：

**谁先用，先解决什么问题，系统第一次怎么工作，第一次协作后学到什么，第二次为什么会更好。**

---

# 1. MVP 目标

ClawLab 的第一版，不是“全自动科研助手”，而是：

> 帮一个研究生基于 CV 和当前课题快速建档，围绕一个具体研究项目完成一次真实任务协作，并把用户修改沉淀成可复用资产。

这个定义里有四个关键词：

### 建档

系统先理解这个人是谁。

### 项目

系统不是按聊天轮次工作，而是围绕一个课题工作。

### 真实任务

不是 demo 问答，而是完成一个用户真会做的任务。

### 沉淀

第一次做完后，系统真的学到点东西，下次更像“专属助手”。

---

# 2. MVP 要解决的核心问题

第一版只解决这三个问题：

### 问题 1

研究者每次都要重新解释自己和项目背景。

### 问题 2

通用 AI 能帮忙，但不了解当前项目脉络，输出不够贴。

### 问题 3

用户改了很多东西，但 AI 不会把这些修改转成长期有用的规则。

所以 MVP 的价值不是“更聪明”，而是：

**更懂你、记得住、会积累。**

---

# 3. MVP 的最小闭环

这是最重要的部分。

## Step 1：用户上传 CV

用户第一次进入 ClawLab，上传一份 CV 或简历。

## Step 2：系统生成初始研究者画像

系统自动抽取：

* 研究方向
* 身份阶段
* 技能栈
* 工具链
* 可能的高频任务

## Step 3：系统通过聊天补 3 到 5 个关键问题

比如：

* 你当前最重要的研究项目是什么
* 你近期最想推进的成果是什么
* 你现在最卡在哪一步
* 你手头有哪些材料

## Step 4：系统创建一个主项目空间

生成：

* 项目卡
* 当前目标
* 当前阶段
* 关键材料清单
* 当前卡点

## Step 5：用户提交一个真实任务

MVP 只支持一类高频任务，建议先选：

**“把零散研究材料整理成结构化输出”**

比如：

* 文献综述提纲
* 研究问题拆解
* paper outline
* figure/storyline 草稿
* 组会汇报提纲

## Step 6：系统输出第一版结果

系统调用：

* 画像信息
* 项目上下文
* 用户给的材料
* 默认研究模板

生成一个结构化草稿。

## Step 7：用户修改

用户对输出做编辑、补充、删改。

## Step 8：系统学习并更新资产

系统识别：

* 用户偏好的表达方式
* 用户关心的结构模式
* 用户强调的项目背景
* 用户不喜欢的内容

然后更新：

* 用户画像中的偏好字段
* 项目记忆
* 一个最小规则集
* 一个初始 SOP 或模板

这就是第一版闭环。

# 4. MVP 的场景切口

你现在先切科研方向，那第一版绝对不能覆盖从 idea 到 paper 全链路。

我建议只切一个中段高频任务：

## 最推荐的首个场景

**研究材料整理与写作前置协作**

具体说，就是用户给你：

* 一些想法
* 几篇论文
* 一些笔记
* 一些分析结果说明

ClawLab 帮他整理成：

* 研究问题结构
* 文献综述提纲
* 论文提纲
* story line 草稿
* 组会汇报草稿

为什么先切这个场景：

### 高频

研究生一直在做。

### 输入相对稳定

CV、项目描述、几份材料就够。

### 输出明确

提纲、草稿、结构化说明都很清晰。

### 适合学习

用户改结构和措辞的过程很容易转成规则。

### 不依赖复杂执行环境

不需要第一版就接代码运行、服务器、实验流程。

---

# 5. MVP 用户画像

第一版用户要更聚焦。

## 核心用户

* 在读博士生或硕士生
* 有明确研究方向
* 正在推进至少一个具体课题
* 已经积累一定材料
* 有频繁写作/整理/汇报需求

## 暂不优先

* 完全没有明确课题的新手
* 强依赖复杂实验流程的湿实验用户
* 希望 AI 直接替代科研决策的人
* 期待一键自动写完整 paper 的用户

---

# 6. MVP 的产品承诺

第一版只承诺三件事：

### 1

我能先理解你，而不是每次从零开始。

### 2

我能围绕一个项目持续协作，而不是只做单轮对话。

### 3

我会从你的修改中学习，让后续输出更贴合你。

不要承诺：

* 自动科研
* 自动 paper 完稿
* 自动实验设计
* 自动端到端代码执行

---

# 7. MVP 核心对象

第一版只做 4 个核心对象。

## 1. Researcher Profile

表示这个用户是谁。

字段建议：

* name
* role
* discipline
* subfield
* methods
* tools
* common_tasks
* writing_preferences
* collaboration_preferences

## 2. Project Card

表示当前主项目。

字段建议：

* project_title
* research_question
* current_goal
* current_stage
* blockers
* materials
* next_step

## 3. Task Card

表示一次具体任务。

字段建议：

* task_type
* input_materials
* expected_output
* related_project
* generated_draft
* user_feedback

## 4. Reusable Asset

表示学习后的沉淀。

字段建议：

* asset_type
* scope
* content
* source
* confidence

`asset_type` 可以先只支持：

* style_rule
* structure_template
* project_note
* sop_seed

# 8. MVP 的 Agent 设计

第一版不用做 7 个独立 Agent，太重。
建议收缩成 **3 Agent 架构**。

## Agent 1：Intake & Orchestrator

职责：

* 解析 CV
* 跟用户聊天补关键信息
* 创建 profile 和 project
* 判断这次任务怎么走

它相当于入口编排器。

## Agent 2：Research Draft Agent

职责：

* 读取项目上下文和任务材料
* 生成结构化输出
* 使用预设学术模板
* 组织成用户需要的草稿

它是第一版的主干生产力 Agent。

## Agent 3：Learning Agent

职责：

* 比较用户修改前后差异
* 提炼偏好和结构规则
* 更新 profile/project/assets

它决定第二次为什么会更好。

这 3 个就够形成闭环。

---

# 9. MVP 的记忆分层

第一版最容易做坏的地方，就是把所有东西都混成“聊天历史”。

所以必须一开始就分层。

## 层 1：Session Context

当前任务临时上下文。

## 层 2：Profile Memory

用户长期信息：

* 方向
* 工具
* 风格
* 偏好

## 层 3：Project Memory

项目相关信息：

* 目标
* 背景
* 材料
* 当前阶段
* 卡点

## 层 4：Asset Store

从任务中学到的规则和模板：

* 偏好规则
* 结构模板
* SOP 种子

第一版哪怕很简陋，也必须逻辑上分开。

# 10. MVP 第一次使用流程

我给你直接写成产品流程。

## 入口

用户打开 ClawLab

### 页面 / 对话 1

系统说：

* 上传你的 CV，或者直接告诉我你是谁、你在研究什么

### 动作 1

用户上传 CV

### 系统动作

自动解析 CV，生成初始理解：

例如：

* 计算生物博士生
* 熟悉 Python / R / Linux
* 高概率需要文献整理、分析叙事、论文写作协作

### 页面 / 对话 2

系统追问少量关键问题：

* 你现在最想推进哪个项目
* 这个项目现在最卡哪一步
* 你手头有哪些材料可以给我

### 动作 2

用户回答，并上传几份材料

### 系统动作

创建：

* Researcher Profile
* Main Project Card

### 页面 / 对话 3

系统给出一个明确提议：

* 我已经建立了你的研究档案和当前项目草图。现在我可以先帮你做一件事：把这些材料整理成文献综述提纲 / paper outline / 组会提纲。你先选一个。

### 动作 3

用户选择一个任务

### 系统动作

生成第一版草稿

### 页面 / 对话 4

用户编辑并反馈

### 系统动作

提炼出：

* 1 到 3 条风格规则
* 1 个结构模板
* 1 个项目记忆更新

闭环完成。

# 11. MVP 的首个任务类型

我建议第一版先只支持 **2 种任务类型**，不要更多。

## 任务类型 A

**文献/想法/笔记 → 结构化研究提纲**

输出可以是：

* 综述提纲
* 研究问题拆解
* 相关工作框架

## 任务类型 B

**项目材料/结果说明 → 写作草稿**

输出可以是：

* paper outline
* figure storyline
* 组会汇报提纲
* 某一节的草稿结构

这样就够了。

为什么不是只支持 1 种？
因为 2 种刚好覆盖：

* “读和想”
* “写和讲”

这已经很接近研究生高频真实使用。

---

# 12. MVP 自动沉淀什么

第一版不要幻想自动生成复杂 Skills。
先沉淀最简单但最有价值的资产。

## 先沉淀 4 类

### 1. Writing Style Rules

例如：

* 更喜欢先讲问题再讲方法
* 避免夸张表述
* 保持严谨语气
* 偏好短段落与清晰层次

### 2. Structure Templates

例如：

* 文献综述提纲模板
* 组会汇报结构模板
* paper outline 骨架

### 3. Project Notes

例如：

* 当前项目目标
* 当前主要研究问题
* 当前最重要材料
* 当前卡点

### 4. SOP Seed

例如：

* “从零散论文到综述提纲”的步骤骨架
* “从分析结果到汇报结构”的步骤骨架

注意是 seed，不是成熟 SOP。

# 13. MVP 需要哪些输入

第一版只需要三类输入。

## 必需输入

### 1. CV / 简历

用于冷启动建档。

### 2. 当前项目描述

一句话到一段话都行。

### 3. 一批任务材料

比如：

* 几篇论文摘要
* 一些研究笔记
* 一段结果说明
* 一个已有提纲草稿

## 可选输入

* GitHub
* Scholar
* 个人主页
* 过往论文或汇报

这些可以留到后面加。

# 14. MVP 输出什么

第一版输出也要克制。

## 用户可见输出

### 1. 研究者画像摘要

例如：

* 你的方向
* 你的技能栈
* 我理解你主要在做什么

### 2. 当前项目卡

例如：

* 项目主题
* 当前目标
* 当前阶段
* 关键卡点

### 3. 一个真实任务的草稿结果

比如综述提纲或 paper outline。

### 4. 学习摘要

例如：

* 我学到了你偏好怎样的结构
* 我会记住这次项目背景
* 下次我会优先用这个模板

这会让用户明显感受到“系统在变得更懂我”。

# 15. MVP 成功标准

我们需要非常具体。

## 产品成功标准

### 1

用户能在第一次会话中完成建档并交付一个真实任务。

### 2

系统生成的结果能让用户觉得“这不是通用 AI 的回答”。

### 3

用户修改后，系统能提炼出至少 2 条有用规则或模板片段。

### 4

用户愿意在第二次同一项目任务中继续使用。

## 体验成功标准

### 1

首次配置不超过 10 分钟。

### 2

追问不超过 5 个关键问题。

### 3

用户能清楚看到系统记住了什么、学到了什么。

---

# 16. MVP 技术与产品边界

第一版必须硬性收边界。

## 不做

* 自动执行代码
* 复杂多文件知识库搜索
* 多项目并行管理
* 自动联网读大量文献
* 真正独立打包/安装 Skill 系统
* 高复杂度 agent swarm

## 只做

* CV 解析
* 对话补全
* 单项目建模
* 单任务协作
* 修改学习
* 最小资产沉淀

---

# 17. MVP 的一句话定义

我建议把 ClawLab v0.1 定义成：

> 一个面向研究生的研究协作助手。它从 CV 和项目描述开始，自动建立研究者档案和项目上下文，帮助用户完成一次真实的研究整理或写作任务，并从反馈中沉淀个人偏好、模板和 SOP 种子。
