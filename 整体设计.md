# 项目整体方案

## 1. 项目本质

这不是一个普通的“科研问答助手”，也不是一个“自动写论文工具”。

它的本质是：

> 一个面向研究者的多 Agent 助手系统。它先通过简历、聊天和项目材料理解“你是谁、你在做什么、你怎么做事”，再围绕你的真实研究任务持续协作，并把过程中沉淀下来的知识、流程、偏好和反馈，逐步转化为可复用的 SOP、Skills 和研究资产。

一句更短的版本：

> 这是一个会先理解研究者，再逐步长成其专属研究操作系统的多 Agent 助手。

---

## 2. 先聚焦的用户

首批目标用户不是泛知识工作者，而是：

* 在读研究生，尤其是博士生
* 有明确研究方向和长期课题
* 已经积累了一些材料，但很分散
* 高频进行文献阅读、想法整理、分析设计、代码协作、论文写作、汇报准备
* 明显感受到“重复解释背景、重复组织材料、重复推进流程”的痛苦

你自己的画像其实就是第一批典型用户：

* 计算生物方向博士生
* 有研究主题
* 有代码、论文、想法、项目材料
* 希望助手越来越懂自己，而不是每次从零开始

---

## 3. 用户核心痛点

这个系统要解决的不是单点能力，而是一组连续性问题。

### 当前研究者常见问题

* 资料很多，但分散，难复用
* AI 能回答问题，但不了解自己
* 每次都要重复解释研究背景
* 同类任务反复做，但流程没有沉淀
* 想法、分析、结果、写作之间断裂
* 用户对 AI 的修改很多，但这些修改不会被系统长期学习
* 没有一个系统能把“用户是谁、项目是什么、如何做事”统一起来

### 本项目瞄准的问题

* 如何让系统先理解研究者
* 如何围绕真实课题建立长期记忆
* 如何把协作过程中的经验沉淀成可复用资产
* 如何让助手从“会回答”升级成“会持续帮你推进研究”

---

## 4. 核心产品目标

第一阶段不是追求“自动科研”，而是追求：

### 目标 1：减少重复解释

用户不用每次都重新说：

* 自己是谁
* 自己做什么方向
* 当前项目背景是什么
* 自己偏好的表达和工作方式是什么

### 目标 2：围绕项目持续协作

系统不是一轮一轮对话，而是围绕：

* 当前课题
* 当前 paper
* 当前分析任务
* 当前项目目标
  持续工作

### 目标 3：形成可复用 SOP

系统在协作中不断沉淀：

* 文献调研流程
* 分析任务启动流程
* 结果整理流程
* 论文写作流程
* 汇报准备流程

### 目标 4：逐步沉淀用户专属 Skills

这些 Skills 不是预设死的，而是从用户真实任务中长出来的：

* 风格类 skill
* 文献整理类 skill
* 论文结构类 skill
* 项目推进类 skill
* 代码说明类 skill

### 目标 5：先形成最小可闭环系统

第一版重点不是“大而全”，而是先跑通：
**建档 → 配置 → 第一次协作 → 用户修改 → 系统学习 → 更新资产**

---

## 5. 产品定位

### 对外定位

> 一个面向研究者的多 Agent 助手，能够根据你的简历、研究方向和项目上下文自动建档，围绕你的真实课题持续协作，并逐步沉淀为你的专属研究操作系统。

### 对内定位

> 一个“研究者建档 + 项目记忆 + 技能沉淀 + 多 Agent 协作”的系统。

---

## 6. 为什么先切科研方向

科研方向适合作为第一阶段，不是因为它最简单，而是因为它有非常强的需求密度。

### 优势

* 用户有明确身份和长期课题
* 用户积累了大量高价值材料
* 用户任务链长，特别需要上下文连续性
* 用户对“越来越懂我”的助手有真实需求
* 用户对知识资产化和 SOP 沉淀的价值感知强

### 风险

* 场景容易做得过重
* 全链路科研闭环太长
* 一开始不能承诺“自动完成科研”

所以第一版必须切得非常克制。

---

## 7. 第一版不做什么

为了保证能落地，第一版明确不做这些：

* 不做全自动从 idea 到 paper 的完整闭环
* 不做自动实验设计与执行
* 不做复杂环境下的自动代码运行代理
* 不做庞大的通用知识库平台
* 不做一开始就要求用户手动配置大量规则
* 不做所有任务都自动生成 Skill

第一版重点是：
**让系统先理解研究者，并围绕一个真实项目开始协作和学习。**

# 核心系统结构

## 8. 系统整体分层

整个系统可以分成五层。

### 第一层：研究者建档层

负责理解用户是谁。

输入：

* CV / 简历
* 个人主页 / Scholar / GitHub
* 聊天信息

输出：

* 用户研究画像
* 初始技能栈
* 可能的任务类型
* 研究方向标签
* 初始协作策略

### 第二层：项目建模层

负责理解用户当前在做什么。

输入：

* 用户当前课题描述
* 论文、草稿、代码、笔记、数据说明
* 聊天中的任务目标

输出：

* 项目卡
* 当前阶段
* 核心目标
* 已有材料
* 未解决问题
* 下一步任务列表

### 第三层：多 Agent 协作层

负责具体执行任务。

包括：

* 用户理解与编排
* 检索和证据整合
* 草稿生成
* 审校与质检
* 学习与沉淀

### 第四层：资产沉淀层

负责把长期有价值的内容抽取出来。

沉淀对象包括：

* SOP
* Skills
* 模板
* 风格规则
* 项目经验
* 常见任务流程

### 第五层：反馈进化层

负责根据真实协作更新系统。

输入：

* 用户修改痕迹
* 用户偏好表达
* 任务完成情况

输出：

* 更新后的画像
* 更新后的项目记忆
* 更新后的 SOP / Skills

---

## 9. 三个核心对象

整个系统里，最重要的不是“对话”，而是三个对象。

### 1. Researcher Profile

表示这个研究者是谁。

包括：

* 学科方向
* 研究身份
* 方法栈
* 工具链
* 写作偏好
* 工作风格
* 质量标准
* 协作偏好

### 2. Project Memory

表示这个研究者当前在做的某个项目。

包括：

* 项目目标
* 研究问题
* 当前阶段
* 关键材料
* 历史决策
* 待办事项
* 风险点
* 项目专属规则

### 3. Skill / SOP Assets

表示系统已经沉淀出的可复用能力。

包括：

* 任务流程
* 输出模板
* 风格规则
* 审校 checklist
* 常用分析路径
* 工具调用规范

这三个对象不能混。

# 首次使用流程

## 10. 第一阶段：研究者首次建档

用户第一次进入系统，不应该像填表，而应该像交给助理一些背景材料。

### 用户可以做的事情

* 上传简历 / CV
* 贴 Scholar / GitHub / 个人主页
* 直接描述最近在做的研究

### 系统自动做的事情

#### 1. 解析身份信息

抽取：

* 教育背景
* 研究方向
* 项目经历
* 论文主题
* 技术栈
* 工具链线索

#### 2. 生成初始研究者画像

例如：

* 计算生物博士生
* 熟悉 Python / R / Linux
* 可能高频任务是文献、分析、写作、汇报

#### 3. 通过聊天补最关键缺口

只追问少量问题：

* 当前最重要的项目是什么
* 目前卡在哪个阶段
* 近期最想推进什么成果
* 手头已有材料有哪些

#### 4. 生成初始配置

包括：

* 研究者卡
* 当前项目卡
* 预装任务模板
* 初始协作模式
* 初始 SOP 种子

---

## 11. 首次配置后，系统应该自动生成什么

不是只生成“个人画像”，而是至少生成这五类对象。

### A. 研究者卡

表示你是谁。

字段示例：

* 学科
* 子方向
* 研究身份
* 方法栈
* 常用工具
* 常见任务
* 写作/协作偏好

### B. 当前主项目卡

表示你现在最重要的研究项目。

字段示例：

* 项目名称
* 核心问题
* 目标成果
* 当前阶段
* 已有材料
* 当前卡点
* 下一步

### C. 初始能力包

为用户自动启用相关任务支持。

例如对计算生物用户：

* 文献综述辅助
* 分析方案讨论
* figure 故事线整理
* 论文提纲生成
* 汇报材料整理

### D. 初始 SOP 种子

例如：

* 新课题文献调研 SOP
* 新分析任务启动 SOP
* 从结果到 figure caption SOP
* 从零散想法到论文提纲 SOP

### E. 协作策略配置

系统默认如何和用户合作。

例如：

* 遇到研究结论时区分事实和推测
* 输出时尽量附上下一步建议
* 新任务默认尝试关联到已有项目
* 优先保持学术严谨和逻辑清晰

# 多 Agent 设计

## 12. 为什么需要多 Agent

因为这个系统不是单轮问答，而是涉及：

* 建模用户
* 管项目
* 检索材料
* 生成内容
* 审校质量
* 学习用户反馈

这些职责最好分开。

---

## 13. 建议的 Agent 角色

### 1. Orchestrator

任务：

* 理解用户目标
* 判断任务属于哪个项目
* 决定调用哪些 Agent
* 组织整体执行顺序

### 2. Research Profile Agent

任务：

* 从简历、聊天和材料中抽取用户画像
* 更新研究者档案
* 识别长期偏好和协作模式

### 3. Project Memory Agent

任务：

* 维护项目状态
* 管理项目材料、决策、任务、阶段
* 保证后续任务都有上下文

### 4. Retrieval Agent

任务：

* 从用户材料、项目资料、外部知识中检索相关信息
* 返回证据和上下文

### 5. Synthesis Agent

任务：

* 根据检索到的内容生成草稿、提纲、分析建议、说明文本

### 6. Critic / Review Agent

任务：

* 检查逻辑、事实、风格一致性
* 标出不确定性
* 做学术表达和结构质量检查

### 7. Learning Agent

任务：

* 比较用户修改前后差异
* 判断哪些应写入长期偏好
* 判断哪些应升级成 SOP 或 Skill

第一版可以合并部分角色，不必全做成独立模块。

# 记忆与资产

## 14. 记忆必须分层

不能把所有东西都叫 memory。

### 1. Session Context

当前对话和当前任务的临时上下文

### 2. Project Memory

某个研究项目专属的持续上下文

### 3. Researcher Profile

用户长期稳定的身份、能力、偏好、风格

### 4. Reusable Assets

从协作中提炼出的 SOP、Skills、模板、规则

真正值钱的是第 4 层。

---

## 15. 资产化对象

系统最终要沉淀的不是原始资料本身，而是这些更高价值的东西：

### SOP

例如：

* 文献整理流程
* 组会准备流程
* 写作推进流程
* 审稿意见处理流程

### Skills

例如：

* 文献对比总结 skill
* paper outline skill
* figure 叙事 skill
* 回复审稿意见 skill

### Templates

例如：

* 论文结构模板
* 汇报模板
* 实验记录模板
* 研究日志模板

### Rules

例如：

* 写作风格规则
* 输出格式规则
* 审稿式检查规则
* 术语使用规则

### Domain Packs

针对方向的术语、工具、分析思路参考包

# 最小闭环

## 16. 第一版最小闭环定义

必须跑通的闭环是：

### Step 1

用户上传 CV，并描述当前研究课题

### Step 2

系统自动生成研究者画像 + 当前项目卡

### Step 3

系统通过少量聊天补关键缺口

### Step 4

系统预装相关能力包和 SOP 种子

### Step 5

用户立刻交给系统一个真实任务
例如：

* 整理一个研究想法
* 做文献综述提纲
* 把分析结果整理成论文段落
* 准备一次组会汇报

### Step 6

系统完成任务草稿

### Step 7

用户修改结果

### Step 8

Learning Agent 分析修改，更新：

* 用户画像
* 项目记忆
* 风格规则
* SOP / Skill 资产

这就是第一版闭环。

---

## 17. 第一版的成功标准

不要用太宏大的指标，先看这些：

### 用户层面

* 用户是否明显少解释背景
* 系统是否能记住项目上下文
* 输出是否比通用 AI 更贴近用户
* 用户是否愿意在同一项目上继续使用它

### 系统层面

* 是否能从 CV 和聊天中建出一个可用的研究者画像
* 是否能围绕单个项目持续工作
* 是否能从至少一次修改中提炼出有用规则
* 是否能生成至少一个后续可复用的 SOP 或 Skill

# 第一版任务范围

## 18. 推荐先支持的任务

不要做完整科研链路，先做中间高频部分。

### 最推荐的首批任务

1. 文献调研与综述提纲
2. 零散研究想法整理
3. 研究问题拆解
4. 分析结果转论文段落 / figure caption
5. 组会汇报材料整理
6. 论文提纲和草稿协作

这些任务的共同点：

* 高频
* 材料明确
* 有输出物
* 容易接收用户反馈
* 容易沉淀为 SOP

---

## 19. 暂缓的任务

第一版先不碰：

* 自动实验执行
* 自动调用复杂外部算力环境
* 自动写整篇 paper 成稿
* 高风险事实生成型任务
* 复杂多项目并行调度

---

# 产品交互原则

## 20. 交互风格原则

这个系统的入口必须像“聪明助理”，而不是“配置表单”。

### 原则 1：少问，问关键

系统先基于简历和材料做最大程度推断，只补关键缺口。

### 原则 2：用聊天套信息，不用问卷压用户

例如：

* 你最近最想推进哪个项目？
* 这个项目现在最卡的是哪一步？
* 你手头有哪些材料我可以先读？

### 原则 3：先让用户马上获得价值

不要先配置半天。
建档后应立刻邀请用户做第一件真实任务。

### 原则 4：在协作中学习，而不是要求用户先定义全部规则

偏好、风格、SOP 应主要从真实使用中长出来。

# 后续演进方向

## 21. 第二阶段可以做什么

等第一版闭环跑通后，再扩展：

### 方向 1：更强的项目长期记忆

* 多项目管理
* 项目间关系
* 时间线追踪

### 方向 2：更强的 skill 自动生成

* 从高频任务中自动提炼 skill
* skill 版本管理
* skill 推荐与复用

### 方向 3：更强的工具接入

* 代码仓库
* 文献管理工具
* 写作工具
* 笔记系统

### 方向 4：更强的科研专属 agent

* 文献 agent
* 写作 agent
* figure/story agent
* 审稿回复 agent

# 最终一句话总结

## 22. 项目总纲

> 我们要做的是一个面向研究者的多 Agent 助手。它先通过简历、聊天和项目材料为研究者自动建档，形成研究者画像、项目记忆和初始能力配置；随后围绕真实研究任务持续协作，并把用户的偏好、流程和修改痕迹逐步沉淀成 SOP、Skills 和长期研究资产，最终演化为一个专属的研究操作系统。
